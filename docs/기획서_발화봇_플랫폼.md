# 발화봇 플랫폼 기획서

---

## 1. 서비스 컨셉

### 한 줄 정의
봇이 소식을 전달하고, 유저가 답글로 참여하는 피드형 커뮤니티.

### 해결하려는 문제
- 커뮤니티에서 원글 작성 허들이 높음
- 글이 없으면 콘텐츠가 안 쌓이고, 콘텐츠가 없으면 유저가 안 옴 (콜드스타트)
- 발화봇이 매일 글을 올려서 답글로 참여하는 구조를 만듦

### 타겟 유저
30~50대 남성. 관심사는 있지만 직접 글 쓸 정도는 아닌 사람. 경기 끝나고, 장 마감 후, 방송 끝나고 한마디 하고 싶은 사람.

## 2. 봇 설계 원칙

### 봇의 역할
| O (한다) | X (안 한다) |
|----------|-------------|
| 소식/팩트 전달 | 특정 편 들기 |
| 논쟁적 팩트 선별 | 가치판단/추천 |
| 화두 던지기 (질문) | 유저 답글에 재답글 |

### 핵심 원칙: Personality through selection
의견을 말하지 않고, **논쟁이 될 팩트를 골라서 던지는 것**이 봇의 캐릭터.
- (나쁨) "삼성전자 살 만합니다" ← 가치판단
- (좋음) "삼성전자 -2.4%, 근데 외국인은 1조 순매수입니다" ← 팩트인데 논쟁적

### 봇 캐릭터
해설위원이 커뮤니티에 눌러앉은 느낌.
```
냉정한 통신사 ──── 해설위원 ──── 방구석 전문가 ──── 밈 봇
                    ↑ 여기
```

### 에너지 스케일링
| 이벤트 크기 | 톤 | 예시 |
|------------|-----|------|
| 일상 | 차분 | "두산 5-3 승. 홈 3연승입니다." |
| 눈에 띄는 활약 | 살짝 올림 | "시즌 20호! 6경기 연속 안타 행진 중." |
| 역사적 순간 | 최대 에너지 | "한 경기 3홈런! 2019시즌 이후 첫 기록!" |

### 팩트 선별 기준
3속성 중 2개 이상 해당하는 팩트를 선택:
- **의외성**: 예상과 다른 사실 (순위 갭, 기록 갱신)
- **긴장감**: 충돌하는 팩트 2개 (발언↔수치, 리드→역전)
- **관여도**: 한마디 하고 싶어지는 사실

### 발화 포맷 (같은 형식 반복 금지)
| 포맷 | 빈도 | 예시 |
|------|------|------|
| 이항선택 | 가장 자주 | "불펜 문제야, 수비 문제야?" |
| 빈칸채우기 | 주 1-2회 | "올 시즌 MVP는 ___" |
| 순위 도전 | 주 1회 | "올 시즌 선발 TOP 3 만들어봐" |
| 예측 | 이벤트 전 | "오늘 스코어 맞춰봐" |
| 반론 유도 | 주 1회 | "이거 반박 가능?" |

## 3. 카테고리별 스펙

### 3-1. KBO 야구

**의도**: 경기 끝나고 한마디 하고 싶은 야구팬의 참여를 유도. 매일 5경기씩 열리므로 발화 빈도 높음.

**데이터 소스**
| 소스 | 수집 방법 | 내용 |
|------|----------|------|
| Daum 스포츠 | Playwright | 경기결과 (팀, 스코어, 구장) |
| Daum 뉴스 검색 | requests | 경기별 관련 기사 (날짜 필터) |

**발화 타이밍**: 경기 종료 후 (~22:00)

**톤**: 해설위원 70% / 흥분하는 팬 20% / 커뮤니티 10%

**문체**: ~입니다(60%) + ~인데요(25%) + ~죠(10%) + ~!(5%)

**발화 예시**
```
KIA 2 : 7 LG
오선우 홈런 터졌지만 팀은 2-7 완패, 시즌 1승 3패입니다. 
홈런 치고도 5점 차로 지는 건 타선 문제인가요, 투수력 문제인가요?
```
```
두산 3 : 13 삼성
삼성 역대 최초 3000승, 강민호 역대 최초 2500경기 출장. 
같은 날 두 개의 '최초'가 한꺼번에 터졌습니다! 
오늘의 주인공, 팀입니까 선수입니까?
```

**30-50대 남성 트리거**
- 역대 기록/순위 ("역대 3번째", "시즌 최초")
- 세대 비교 ("박찬호 이후 최초")
- 상반된 팩트 병치 ("타율 1위인데 팀 순위 8위")

### 3-2. 주식

**의도**: 장 마감 후 "오늘 시장 어땠냐" 얘기하고 싶은 개인투자자의 참여를 유도. 대형주 + 이슈주 2가지 테마로 운영.

**데이터 소스**
| 소스 | 수집 방법 | 내용 |
|------|----------|------|
| Daum 금융 시총 상위 | Playwright | 시총 Top 10 중 등락 큰 2종목 (대형주) |
| Daum 금융 상승률 | Playwright | 시총 상위와 안 겹치는 이슈주 2종목 |
| Daum 뉴스 검색 | requests | 종목별 뉴스 |

**두 가지 테마**
| 테마 | 선정 기준 | 역할 |
|------|----------|------|
| 대형주 | 시총 상위 10 중 등락률 큰 순 | 다 아는 종목. 시장 흐름과 엮어서 발화 |
| 이슈주 | 상승률 상위 중 시총 상위 아닌 것 | 오늘 뭐가 핫했는지. 뉴스 맥락 포함 |

**발화 타이밍**: 장 마감 후 (15:30~16:00)

**톤**: 해설위원 80% / 흥분 10% / 커뮤니티 10% (KBO보다 차분)

**금지사항**: 매수/매도 추천 절대 금지. 주식은 돈이 걸려있어서 과한 캐릭터가 신뢰를 깎음.

**발화 예시 — 대형주**
```
삼성전자 -2.43%
코스피가 1.4% 오르며 5,858까지 갔는데 삼성전자는 혼자 역행했습니다. 
외국인이 1조 1천억 순매수로 지수를 끌어올렸는데, 그 돈은 대체 어디로 간 건가요?
```

**발화 예시 — 이슈주**
```
주성코퍼레이션 +29.97%
거래량 481만주 실으면서 상한가 턱밑까지 갔습니다. 
이 거래량이면 내일도 이어가는 건가요, 오늘 다 실은 건가요?
```

**논쟁적 팩트 유형 (주식 특화)**
- "실적 어닝서프라이즈인데 주가 하락"
- "증권사 목표가 상향 vs 실제 주가 하락"
- "외국인 매도 vs 기관 매수"
- "반도체 전멸인데 2차전지만 상승"

### 3-3. 방송 — 무명전설

**의도**: 매주 수요일 방영 후 시청자들의 참여 유도. 트로트 서바이벌 특성상 탈락/합격에 대한 의견이 갈리기 쉬움.

**프로그램 정보**
- MBN 트로트 서바이벌, 매주 수요일 방영
- MC: 김대호, 장민호 / 심사: 남진, 조항조, 주현미 등
- 99명 도전자 → TOP5 → 최종 우승

**데이터 소스**
| 소스 | 수집 방법 | 내용 |
|------|----------|------|
| Daum 뉴스 검색 | requests | 방영 후 관련 기사 (날짜 필터) |

**발화 타이밍**: 방영 다음날 오전

**톤**: 해설위원 60% / 흥분 25% / 커뮤니티 15% (가장 캐주얼)

**발화 예시**
```
무명전설 7회 - 이창민 본선 3차 직행
이창민이 성리를 꺾고 본선 3차 직행했다. 
신성도 데스매치에서 10:5 완승하면서 유명 가수들이 본격적으로 치고 올라오는데, 
무명 참가자들이 이 벽을 넘을 수 있긴 한 건가?
```

**논쟁적 팩트 유형 (방송 특화)**
- "심사위원 3:2로 갈린 탈락"
- "시청률 1위인데 전주보다 하락"
- "심사위원이 '역대 최고'라고 했는데 시청자 반응은?"

## 4. 시스템 구조

### 파이프라인
```
[소스 수집] → [팩트 선별] → [발화 생성] → [피드 게시]
                                ↑
                         [피드백 루프] (향후)
                     이전 발화의 답글 맥락
```

### 멀티에이전트 구조
```
[오케스트레이터] — 컨트롤타워, 직접 로직 없음
    │
    ├─ [크롤러 에이전트] — 카테고리별 독립 실행
    │   ├── kbo_crawler.py
    │   ├── stock_crawler.py
    │   └── broadcast_crawler.py
    │
    ├─ [작성 에이전트] — 카테고리별 독립 실행
    │   ├── kbo_writer.py
    │   ├── stock_writer.py
    │   └── broadcast_writer.py
    │
    └─ [피드 에이전트] — UI/답글/좋아요 관리
        └── feed_agent.py
```

- 에이전트 간 통신: 공유 DB (SQLite)
- 각 에이전트 독립 실행 가능 (타이밍 분리)
- 카테고리 추가 시: 크롤러 + 작성 에이전트 + yaml 설정만 추가

### LLM 전략
- **1순위**: Claude Code CLI (로컬 실행, 무료, 최고 퀄리티)
- **2순위**: Groq API (llama-3.3-70b, 무료 tier, fallback)
- 배포 서버에서는 LLM 호출 안 함. 로컬에서 생성 → DB 저장 → push

### 카테고리별 설정 구조
```
configs/
├── kbo.yaml                   # KBO 야구
├── stock.yaml                 # 주식
├── broadcast_무명전설.yaml     # 무명전설
├── broadcast_대군부인.yaml     # 21세기 대군부인
└── broadcast_[새프로그램].yaml  # 운영자가 수동 추가
```

## 5. 프로토타입 UI

### 화면 구성
- **사이드바**: 카테고리 선택 → 데이터 수집 (기간) → 발화글 생성 (날짜) → 전체 삭제
- **피드**: 발화글 목록 (경기결과/종목 볼드 + 본문 분리)
- **상세**: 발화글 + 답글 목록 + 답글 작성

### 기술 스택
| 구분 | 기술 |
|------|------|
| UI | Streamlit |
| DB | SQLite |
| 크롤링 | Playwright (동적), requests+BS4 (정적) |
| 배포 | Streamlit Cloud |
| 코드 관리 | GitHub |

## 6. 핵심 지표 (향후)

| 지표 | 의미 |
|------|------|
| 발화당 답글 수 | 봇 글 하나에 답글이 몇 개 달리는가 |
| 재방문율 | 답글 단 유저가 다음날 다시 오는가 |
| 답글 → 원글 전환 | 답글 달던 유저가 직접 글도 쓰기 시작하는가 |

## 7. 향후 과제

| 과제 | 설명 | 우선순위 |
|------|------|:--------:|
| 피드백 루프 | 이전 답글 → 다음 발화 반영 | 높음 (답글 쌓인 후) |
| 전/후 발화 | 경기 전 예측 → 경기 후 결과 (이중 발화) | 중간 |
| 포맷 로테이션 추적 | 직전 포맷 DB 기록 → 다음에 다른 포맷 | 중간 |
| DB 영속성 | SQLite → 외부 DB (서버 재시작 시 데이터 유지) | 중간 |
| 추천 시스템 | 답글 기반 개인화 피드 | 낮음 (데이터 축적 후) |
